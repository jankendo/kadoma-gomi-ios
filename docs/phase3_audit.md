# Phase 3 Audit

## 1. 現在の実装概要

このアプリは門真市限定の SwiftUI iOS アプリです。iOS 17.0 以上、外部依存なし、GitHub Actions の macOS runner で unsigned IPA を生成する構成です。A-F 全地区の基本収集曜日、大倉町1-20 -> A地区プリセット、分割JSONから master/manifest/SHA を生成する仕組みを持ちます。

## 2. 第2フェーズからの到達点

- A-F 全地区の基本曜日と地区選択UIが実装済み。
- `collection_areas.json`、`collection_schedule.json`、`garbage_items.json`、`special_rules.json` から本番 master を生成可能。
- `validate_data.py`、`generate_master.py`、`run_quality_checks.py` による検証が可能。
- 検索正規化、通知予定プレビュー、SwiftUI Preview、GitHub Actions build/pages/source-monitor が整備済み。

## 3. 現在の残課題

- 2026年12月から2027年1月の年末年始停止日・振替日は公式確定情報を未確認。
- 実機通知到達、通知許可拒否後の再許可導線、VoiceOver実走査、Dynamic Type最大、iPhone SE相当表示は実機未検証。
- 検索辞書は100件で日常品目の網羅性が不足。
- Widget / App Shortcuts が同じ読み取りデータを使うための専用層が不足。

## 4. 門真市限定アプリとしての品質リスク

自治体横断の抽象化は入れない方針です。一方で、門真市の年度更新、広報、PDF更新に追従できないと誤案内につながります。マスタの municipalityCode/name、公式URL、A-F地区検証をCIで固定して、門真市以外のデータ混入を検出する必要があります。

## 5. 収集日データのリスク

基本曜日は公式ページから確認済みですが、PDF上の実日付表と完全照合したわけではありません。ホーム、カレンダー、通知は同じ生成ロジックを使うためロジックの一貫性はありますが、公式の臨時変更にはマスタ更新で追従する必要があります。

## 6. 年末年始・例外日データのリスク

A地区PDFで「12月・1月は収集日を変更する場合があります」という注意を確認しました。ただし、2026年度末の確定停止・振替日は確認できていません。未確認日を確定例外に入れると誤通知になるため、Phase 3 では確定例外を空にし、`exceptionRules` の needs_review として扱います。

## 7. 検索辞書のリスク

100件では日常の迷いやすい品目に不足があります。公式ページでカテゴリ根拠が明確な品目と、公式確認が必要な品目を分ける必要があります。未確認品目は `requiresOfficialCheck=true` としてUIに表示します。

## 8. 通知機能のリスク

通知は便利ですが、未確認の年末年始や地区変更後に古い通知が残ると信頼を損ないます。地区変更・マスタ更新時の再設定導線、通知予定プレビュー、自然な日本語文面、未確認期間の注意文が必要です。

## 9. アクセシビリティのリスク

検索結果カードや通知プレビューは情報量が増えるため、小型画面・Dynamic Type最大で縦に伸びても読める構成が必要です。危険物・粗大候補・公式確認推奨は色だけに依存せずバッジ文言で示します。

## 10. リリース前に優先すべき改善

1. 年末年始確定日を入れず、needs_review として安全に扱う。
2. 検索辞書を200件以上へ拡張し、未確認品目を明示する。
3. Widget/App Shortcuts 用の読み取り専用サマリー層を追加する。
4. 通知プレビューと文面を実機確認しやすい形へ強化する。
5. 実機QAチェックリストを整備する。

## 11. 今回の実装方針

門真市限定を維持し、公式確認できた情報だけを確定扱いにします。未確認の年末年始は確定例外にせず、UI・通知・QAで注意喚起できるメタデータに留めます。検索辞書は200件以上に拡張し、危険物・粗大候補・公式確認推奨をデータとUIの両方で表現します。
