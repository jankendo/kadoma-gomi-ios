# Phase 2 Audit

## 1. 現在の実装概要

前回までに、SwiftUI単一ターゲットの門真市ごみ管理アプリとして、5タブ構成、デザインシステム、共通カード、オンボーディング、初期マスタ、GitHub Pages配信、GitHub Actions unsigned IPAビルドが整備されている。

## 2. 前回から改善されている点

- 今日/明日/次回をホームで優先表示している。
- 検索画面に検索欄、カテゴリ絞り込み、検索結果カードがある。
- カレンダーに月表示、リスト表示、凡例がある。
- 設定に地区、通知、データ更新、公式リンク、アプリ情報が整理されている。
- App Icon、LaunchBackground、AccentColorが追加されている。
- 分割JSONが追加され、将来のデータ管理に向けた下地がある。
- GitHub ActionsでビルドとPages配信が成功している。

## 3. 現在も残っている課題

- 分割JSONはA地区だけで、A-F全地区対応ではない。
- 分割JSONから本番masterを生成する公式なスクリプトがない。
- JSONスキーマとデータ品質チェックがCIに組み込まれていない。
- 検索正規化は簡易で、ひらがな/カタカナ、全角/半角、類義語への耐性が不足。
- 通知予定をユーザーが事前確認できない。
- 地区選択UIは住所入力中心で、A-F地区を直接選べない。
- Previewは最低限で、画面サイズ/Dynamic Type/ダークモード/状態別確認が不足。

## 4. データ品質上のリスク

- 年末年始の実例外日は公式PDF/広報の人間レビューが必要。
- 現在のmasterは手作業コピーで、分割JSONとの乖離が起きうる。
- 品目辞書には`confidence`が低い手動初期データがあり、公式確認推奨表示が必要。
- 町名の番地範囲分岐がある地区では、町名だけでは誤判定する可能性がある。

## 5. UI/UX上のリスク

- 小さい画面でカレンダーセル内ラベルが詰まる可能性がある。
- Dynamic Type最大では横並びボタンやチップが折り返し不足になる可能性がある。
- 地区変更時の影響がまだ十分に強調されていない。
- 公式未確認・例外未確認の注意が一部画面で弱い。

## 6. 通知機能上のリスク

- 通知許可状態は表示しているが、通知予定一覧がないため信頼性を確認しづらい。
- 地区変更時に通知を自動再設定する導線が弱い。
- 例外未確認日の通知をどう扱うかが明確ではない。

## 7. カレンダー計算上のリスク

- 週次・第n曜日・specificDatesには対応しているが、検証は手元スクリプト中心でCI化されていない。
- A地区以外のルールが未投入だった。
- 例外日がnote扱いだけで、年末年始変更のcancel/add実データは未確定。

## 8. CI/CD上のリスク

- build workflowはビルドのみで、データ検証やmaster生成検証を実施していない。
- source-monitorは差分検知のみで、master生成とは連動していない。
- Node 20警告はNode 24強制で回避しているが、上流action更新までは警告が残る可能性がある。

## 9. 優先して直すべき項目

1. A-F全地区データを公式確認済みとして分割JSONへ入れる。
2. master生成/manifest SHA生成をスクリプト化する。
3. schema + validate_dataで不整合をCI検出する。
4. 検索正規化と検索テストを強化する。
5. 地区選択UIと地区変更時の通知再設定を明確にする。
6. 通知予定プレビューを追加する。
7. Preview/手順書/検証docsを更新する。

## 10. 今回の実装方針

- 公式HTMLで確認できるA-F地区の町名・曜日ルールを確定データとして採用する。
- 年末年始例外は未確認注意としてUIとデータに残す。
- runtime互換の`initial_master_27223_2026.json`を、分割JSONから生成する運用へ切り替える。
- 外部依存は追加せず、Python標準ライブラリで検証・生成・テストを実装する。
- Swift側は既存デザインシステムを活かし、地区選択・通知予定・検索品質に絞って安全に拡張する。
