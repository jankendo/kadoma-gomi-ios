# UX Upgrade Plan

## 1. 今回のアップグレードの目的

門真市の生活インフラ系アプリとして、毎日使える信頼感、今日/明日の即時理解、分別検索の使いやすさ、地区設定の分かりやすさ、通知の安心感を製品レベルに引き上げる。

## 2. 想定ユーザー

- 門真市在住者
- 大倉町1-20周辺のA地区利用者
- 引っ越してきたばかりの人
- 高齢者、スマホ操作が苦手な人
- 忙しい世帯、ごみ出しを忘れやすい人
- 分別が苦手な人

## 3. 最重要ユーザーストーリー

- アプリを開いた瞬間に今日出せるごみを確認できる。
- 明日出すごみを確認できる。
- ごみ名を検索して分別方法を確認できる。
- 自分の地区を設定できる。
- 大倉町1-20に対応する地区設定を使える。
- 収集日前日に通知を受け取れる。
- 分別に迷ったら短い説明と公式情報への導線を見られる。
- データがない場合でも次に何をすればよいか分かる。
- エラー時に原因と対処が分かる。

## 4. 現状の問題点

- 画面の情報優先度が弱く、今日の行動が最上位に固定されていない。
- 検索が単純で、カテゴリ絞り込みや空状態が弱い。
- 設定が機能別に整理されていない。
- 通知許可状態と再設定結果が分かりづらい。
- 初回起動で何をすれば良いか案内がない。
- アセット、デザインシステム、共通状態UIが不足。

## 5. 改善後の理想体験

起動直後、上部に地区と更新状態、中央に今日のごみカード、下に明日・次回・検索導線が並ぶ。検索では品目名を入力するとカテゴリと捨て方がカードで出る。設定では住所/通知/データ更新/公式情報が明確に分かれ、エラー時も次の行動が示される。

## 6. 情報設計

- Home: 今日、明日、検索、直近予定、注意、更新状況
- Calendar: 月表示、直近予定、凡例、例外注意
- Search: 検索、カテゴリ、候補、結果、公式確認導線
- BulkyWaste: 粗大ごみの定義、予約、電話/Web、処理券
- Settings: 地区、通知、データ、公式情報、使い方、アプリ情報

## 7. 画面構成

既存5タブを維持する。オンボーディングは初回のみフルスクリーンで表示し、設定から再表示可能にする。

## 8. ナビゲーション設計

TabViewをトップレベルの安定ナビゲーションとして維持。ホームのクイック導線はタブ選択を切り替えるだけにして、複雑な深い遷移は作らない。

## 9. デザインシステム方針

セマンティックカラー、余白、角丸、カード、バッジ、ボタン、状態表示を一元化する。公共サービスに近い落ち着いた色を基調に、ごみ種別色は識別目的に限定する。

## 10. アセット方針

著作権不明素材は使わない。アイコンはSF Symbolsを主軸にする。App Iconは自作PNGをAsset Catalogへ追加する。装飾目的の画像は増やさない。

## 11. データ更新方針

既存のmonolithic master JSONをランタイム互換として維持しつつ、カテゴリ/品目/地区/収集ルール/特別ルールの分割JSONを追加し、将来のレビュー更新を容易にする。

## 12. アクセシビリティ方針

Dynamic Typeのテキストスタイルを使う。重要カードはVoiceOverラベル/ヒントを追加する。色だけでなくアイコンとテキストを併用する。タップ領域を44pt以上に保つ。

## 13. ダークモード方針

systemBackground / secondarySystemGroupedBackground系の動的色を中心に使う。固定の淡色背景や低コントラスト文字を避ける。

## 14. 実装対象

- DesignSystem
- App Icon / Asset Catalog
- Onboarding
- Home改善
- Search改善
- Calendar改善
- BulkyWaste改善
- Settings/地区/通知/データ更新改善
- State components
- split data JSON
- documentation

## 15. 実装しないこと

- GPS/地図API
- App Store署名/配布
- PDFの完全自動OCR
- 全地区A-Fの完全UI切替
- WidgetKit本実装
- Apple Watch / Siri / 写真判定

## 16. リスク

- WindowsではXcode/Simulatorが実行できない。
- 通知許可導線は実機での最終確認が必要。
- 手書きpbxprojの更新ミス。
- 分割JSONとmonolithic JSONの二重管理。

## 17. リスク回避策

- GitHub ActionsのmacOS runnerでビルド確認する。
- JSON検証、manifest SHA検証、カレンダールール検証をローカルで行う。
- pbxproj内に全Swift/Asset参照があることをスクリプトで確認する。
- docs/data_model.mdに分割JSONは運用補助、runtimeはmonolithicと明記する。

## 18. 検証方法

- `git diff --check`
- JSON / plist parse
- master SHA検証
- カレンダールール期待値チェック
- pbxproj参照チェック
- GitHub Actions unsigned IPA build
- IPA artifact内部検査
- Pages manifest/master HTTP 200確認
